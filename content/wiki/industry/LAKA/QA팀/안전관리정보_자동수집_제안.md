## 1. 배경 및 목적

### 현행

[[품질관리_RR_2026-05]] 15번 업무 "화장품 안전관리"의 일부로, 김건 책임이 **월 1회 이상** 국내·외 유해사례 정보를 수집·기록한다 (자사 매뉴얼 별지 1호 서식). 수집 출처는 온라인 사이트, 기사 문헌, 학회 등으로 명시되어 있다.

문제는 정보의 양과 속도다:
- **식약처 회수 공고**·**FDA Recalls**·**UK OPSS** 등 관 공식 채널만으로도 일 단위 새 항목 발생
- 2026년 들어 LAKA에서도 직접 영향 사례 발생: [[2026-02-W2_SCM주간업무보고]] **수단레드 잠정 판매중지 사건** (2/3 공문 접수, 2/10 해제). 이 사건은 외부 채널에서 먼저 시그널이 잡혔다면 더 빠른 대응 가능
- 글로벌 채널 확장기 — [[세포라]] US·[[부츠]] UK·[[Qoo10_일본]] 동시 운영 → FDA·OPSS·PMDA 3개 관할 동시 모니터링 필요
- 김건 책임 단독 주관 7건 + 공동 부담당 5건([[품질관리_RR_2026-05]] R&R) → 월 1회 수동 수집은 실질적 정보 누락 위험

### 목표

**월 1회 수동 수집 → 일 단위 실시간 모니터링 + 주간 자동 다이제스트로 격상**. 김건 책임의 작업량은 줄이고 정보 커버리지는 늘린다. [[품질팀_슬랙운영_초안_2026-05]]에서 정의한 채널 구조 활용.

### 비목표

- 식약처 안전성 정기보고([[품질관리_RR_2026-05]] 14번)의 **공식 제출 자체는 자동화 대상 아님** — 별지 1호 서식 초안만 자동 생성, 김건 책임 검수 후 정식 보고
- 클레임 처리 프로세스([[품질관리_RR_2026-05]] 1번) 자체는 자동화 대상 아님 — 사전 시그널 탐지만

## 2. 수집 대상 정리

### 5개 카테고리 × 출처

| # | 카테고리 | 핵심 출처 | 수집 방식 |
|---|---------|----------|----------|
| 1 | **국내 관 공식** | 식약처 의약품안전나라 회수·판매중지 공고 / 식약처 보도자료 / [[대한화장품협회]] KCIA 공지사항 / 한국소비자원 안전경보 / 환경부 환경공단 EPR | RSS 또는 웹 스크래핑 |
| 2 | **해외 관 공식** | FDA Recalls·Import Alerts (53-06 색소·66-40 GMP) / FDA CAERS 공개 DB / UK OPSS Product Safety Database / EU RAPEX Safety Gate (화장품 카테고리) / 일본 PMDA 회수 게시판 | RSS·API (FDA는 공개 API 풍부) |
| 3 | **국내 미디어** | 네이버뉴스·다음뉴스 — 키워드: 화장품 회수·립틴트 부작용·라카·경쟁사·ODM | 네이버 알림·네이버 뉴스 API |
| 4 | **해외 미디어·SNS** | Google News (영문) / X (구 트위터) — K-beauty recall·LAKA cosmetics·Fruity Glam Tint / TikTok #fyp 부정 키워드 | Google Alerts (이메일→Slack), X API, 모니터링 SaaS |
| 5 | **학술 문헌** | PubMed `(cosmetic) AND (adverse OR allergy OR dermatitis)` / Google Scholar Alerts / 대한피부과학회지·대한화장품학회지 | PubMed RSS, Scholar 이메일 |

### 키워드 매트릭스 (Tier 별 즉시 알림 우선순위)

```
🔴 Tier A — LAKA 직접 영향 (즉시 알림)
- "라카 화장품" / "LAKA cosmetics" / "라카코스메틱스"
- 자사 제품명: "프루티 글램 틴트" / "Fruity Glam Tint" / "Bonding Glow Lipstick" 등 핵심 SKU
- 자사 ODM 회수 시그널: [[한국콜마]] / 코스맥스 / 화성코스메틱 / 코스메카 / 씨앤씨

🟡 Tier B — 카테고리·경쟁사 (일일 다이제스트)
- 한국 K-뷰티 립 카테고리 회수·부작용
- 경쟁사 클레임: rom&nd 구순염, BBIA 건조, 페리페라 등 — [[K뷰티_립_CS_벤치마크]] 활용
- 25종 알레르기 향료 부작용 보고 — [[향료_알레르기유발성분_표시]]
- FDA Import Alert 53-06 / 66-40 새 등재 (한국 제조사 적발 시)

🟢 Tier C — 일반 동향 (주간 다이제스트)
- 글로벌 화장품 규제 개정 동향
- 신규 알레르기 유발성분 추가 검토 동향
- 학술 논문 (월간 정리)
```

## 3. 시스템 설계 — 3계층 아키텍처

```
┌─────────────────────────────────────────────┐
│  수집층 (Collection Layer)                  │
│  RSS / API / 웹 스크래퍼 → 원시 데이터 풀  │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  필터·요약층 (Processing Layer)             │
│  Claude API:                                │
│  - LAKA 관련성 분류 (Tier A/B/C)            │
│  - 한 줄 요약 (한국어)                      │
│  - 키워드 매칭으로 자동 라벨링              │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  배포층 (Distribution Layer)                │
│  - Tier A: #qa-시그널-피드 즉시 알림 + 멘션 │
│  - Tier B: #qa-시그널-피드 일일 다이제스트  │
│  - Tier C: 주간 다이제스트 (월요일 09:00)   │
│  - 별지 1호 서식 자동 초안 (월말)           │
└─────────────────────────────────────────────┘
```

### 권장 도구 스택

| 계층 | 1단계 (Quick Win) | 2단계 (확장) |
|------|-------------------|--------------|
| 수집 | Slack RSS 앱 (8개 소스 직구독) | GitHub Actions + Python (RSS·스크래퍼·API 통합) |
| 필터·요약 | (없음 — RSS 원문 그대로) | Claude API (한국어 요약 + Tier 분류) |
| 배포 | Slack 채널 직접 게시 | Slack Webhook + 별지 1호 서식 자동 생성 |
| 비용 | **무료** | GitHub Actions 무료 한도 + Claude API ~5만원/월 |
| 인력 | 김건 책임 셋업 0.5일 | 외부 개발자 또는 사용자 직접 구현 2~3일 |

## 4. 단계별 실행 계획

### 1단계 — Quick Win (2주 내 완료)

**목표**: 김건 책임이 직접 셋업 가능한 수준의 자동 수집을 즉시 시작. 비용 0원.

**작업**:
1. 신규 Slack 채널 `#qa-시그널-피드` 개설 (Public, 품질팀 + 천승범·관심 부서 자율 참여)
2. Slack RSS 앱 활성화 후 다음 8개 RSS 소스 구독:
   - 식약처 의약품안전나라 회수·판매중지 공고 (RSS 있음)
   - 대한화장품협회 공지사항 (RSS 또는 정기 스크랩)
   - FDA Recalls — Cosmetics 카테고리 RSS
   - UK OPSS Product Safety Database RSS
   - EU Safety Gate Cosmetics RSS
   - Google Alerts 이메일→Slack: "라카 화장품" / "LAKA cosmetics" / "K-beauty recall"
   - PubMed RSS: `(cosmetic[Title]) AND (adverse OR allergy OR dermatitis)`
   - 일본 PMDA 화장품 회수 게시판 (RSS 없으면 스크래퍼 필요 — 2단계)
3. 채널 핀고정 메시지로 운영 룰 게시 (피드 게시물 처리 책임자: 김건, 주간 요약 작성자: 김건)

**완료 기준**: 채널에 매일 최소 5건 이상 자동 게시 / 김건 책임 수동 수집 시간 ≥30%↓

### 2단계 — AI 필터·요약 도입 (3개월 내)

**목표**: 단순 RSS 흘러내림 → 한국어 요약 + Tier 분류된 다이제스트로 격상. LAKA 관련성 자동 판정.

**작업**:
1. **자체 구축 vs SaaS 결정** (의사결정 #2)
   - 자체 구축: GitHub Actions + Python 스크립트 + Claude API
   - SaaS: Make.com 또는 Zapier 워크플로
   - 권장: **자체 구축** (LAKA 내부 데이터 안전·확장성·장기 비용 측면 유리)
2. Python 스크립트로 다음 추가 소스 통합:
   - 화해(hwahae)·@cosme·Sephora·Boots 리뷰 부정 키워드 자동 추출
   - 네이버 뉴스 API
   - X (트위터) API — LAKA 브랜드 mention
3. Claude API 호출 워크플로 구축:
   - 각 신규 항목 → Claude에게 "LAKA 관련성 분류 + 한 줄 요약 + Tier 판정" 요청
   - 응답을 Slack 메시지로 포맷 (Tier 색상 이모지 자동 표시)
4. Tier A 발생 시 천승범 본부장 + 이지철 대표 @멘션 (조건: LAKA 브랜드명 직접 언급 OR LAKA ODM 회수 OR FDA Import Alert 한국 신규 등재)

**완료 기준**: 일 평균 다이제스트 30건+ → AI 필터 거친 Tier B+ 5건 이내로 압축 / 김건 책임 검수 시간 ≤15분/일

### 3단계 — 별지 1호 서식 자동화 + SNS 시그널 통합 (6개월 내)

**목표**: 월말 별지 1호 서식 자동 초안 생성. SNS·리뷰 조기 탐지로 LAKA 직접 위기 사전 감지.

**작업**:
1. **별지 1호 서식 자동 초안**: 월간 누적 데이터 → Claude로 별지 1호 서식 항목별 자동 채우기 → 김건 책임이 검수 후 식약처 보고에 사용
   - **완전 자동화 X, hybrid 권장** (의사결정 #5) — 식약처 공식 보고는 사람 검수 단계 필수
2. **SNS 시그널**:
   - TikTok·X·인스타그램에서 LAKA 제품명·해시태그 모니터링
   - 도구: Apify (저비용) 또는 Brand24 (SaaS, 비용 높음)
   - [[2026-02-W2_SCM주간업무보고]] 수단레드 사건처럼 외부 SNS에서 먼저 터지는 경우 대비
3. **경쟁사 클레임 동향**: rom&nd 구순염·BBIA 건조·페리페라 등 ([[K뷰티_립_CS_벤치마크]] 활용) — 경쟁사 부작용 발생 시 LAKA 사용 성분 비교 자동 알림

**완료 기준**: 월말 별지 1호 서식 초안 자동 생성 + 김건 책임 검수 1시간 이내 완료 / SNS 시그널 Tier A 발생 시 30분 이내 알림

## 8. 리스크 및 한계

### 가. 정보 누락 리스크
- RSS·API 없는 사이트(특히 한국 일부 기관) → 스크래퍼 필요, 사이트 구조 변경 시 깨짐
- **완화**: 1단계는 RSS·API 있는 핵심 8개에서 시작, 2단계 스크래퍼는 모니터링 + 사이트 변경 알림 추가

### 나. AI 분류 오류
- Claude가 Tier A를 Tier C로 잘못 분류할 가능성 → 중요 시그널 놓침
- **완화**: 핵심 키워드(LAKA·ODM 회사명 등)는 AI 거치지 않고 **규칙 기반으로 강제 Tier A 처리**. Claude는 보조 분류기로만 활용

### 다. 알림 피로 (Alert Fatigue)
- Tier A·B 알림이 너무 많으면 진짜 중요한 것이 묻힘
- **완화**: 임계치 보수적 설정 + 분기별 임계치 재조정 + 김건 책임이 매주 분류 정확도 검수

### 라. 식약처 보고 자동화 한계
- 별지 1호 서식 완전 자동 작성 → 식약처 보고 → **부정확한 정보가 그대로 정부 제출 위험**
- **완화**: AI는 초안만, 김건 책임이 반드시 검수 단계 (의사결정 #5)

### 마. 보안·내부 데이터 노출
- SaaS(Zapier·Make) 사용 시 LAKA 내부 데이터가 외부 클라우드 경유
- **완화**: 2단계부터는 GitHub Actions(Anthropic 직접 API) 또는 n8n 셀프호스팅 권장. 내부 시그널은 Slack 내부에만 유지

## 9. 의사결정 필요 사항 (본부장 확정 5건)

1. **1단계만 시작 vs. 1+2단계 동시 추진**
   - 권고: **1단계 즉시 시작, 2단계는 1단계 운영 데이터 확인 후 결정** (1주 운영 → 일일 다이제스트 양·정확성 평가 → 필요시 2단계 착수)

2. **자체 구축 vs. SaaS**
   - 권고: **자체 구축 (GitHub Actions + Python + Claude API)** — LAKA 내부 데이터 안전·확장성·장기 비용 측면 유리. 단 2단계 진입 시 결정

3. **Slack 채널 통합 위치**
   - 권고: **별도 신설 `#qa-시그널-피드`** (피드 채널 분리, 조치는 `#qa-인허가-법규`로 이관)

4. **🔴 Tier A 즉시 알림 트리거 조건**
   - 권고: 위 5개 조건 그대로 시행. 1주 운영 후 임계치 재조정

5. **별지 1호 서식 자동 채우기**
   - 권고: **AI 초안 + 김건 책임 검수 Hybrid** — 완전 자동 X. 식약처 공식 보고는 사람 책임
