## 2.2 Vibe Coding vs Human-Agent Engineering

### Vibe Coding이란?

> **Vibe Coding**: 자연어로 대화하듯 코드를 생성하는 접근법. 
> "감(vibe)으로 코딩하기"

```
Vibe Coding 예시:
"로그인 기능 만들어줘" → AI가 전체 코드 생성

장점: 빠른 프로토타이핑, 진입 장벽 낮음
단점: 품질 불일정, 프로덕션 레벨 어려움
```

### Human-Agent Engineering (권장)

> **Human-Agent Engineering**: 개발자가 의도를 설계하고, AI Agent를 관리/감독하는 접근법

```
Human-Agent Engineering:
1. 개발자: 요구사항 분석, 아키텍처 설계
2. AI: 구현 초안 생성
3. 개발자: 코드 리뷰, 품질 검증
4. AI: 피드백 기반 수정
5. 개발자: 최종 승인, 배포

핵심: AI는 "유능하지만 경험 없는 인턴"
      개발자는 "관리자/리뷰어"
```

### CS146S 핵심 철학 재확인

| Vibe Coding | Human-Agent Engineering |
|-------------|------------------------|
| "AI야 해줘" | "이렇게 해줘, 이유는..." |
| 결과만 확인 | 과정과 결과 모두 검증 |
| 빠르지만 불안정 | 느리지만 신뢰할 수 있음 |
| 프로토타입에 적합 | 프로덕션에 적합 |

# 3. Cursor 심화 활용

## 3.1 Cursor의 3가지 핵심 모드

### ① Tab (자동완성)

```
기능: 다음에 올 코드를 예측하여 제안
단축키: Tab으로 수락

특징:
- 320ms 내외의 빠른 응답
- 현재 파일 컨텍스트 기반
- 단일/멀티라인 모두 지원

적합한 상황:
- 반복적인 코드 작성
- 보일러플레이트 생성
- 패턴 기반 코딩
```

### ② Inline Edit (Cmd+K / Ctrl+K)

```
기능: 선택한 코드 영역을 자연어로 수정
단축키: Cmd+K (Mac) / Ctrl+K (Windows)

특징:
- 외과적으로 정밀한 수정
- Diff 미리보기 제공
- 부분 수락 가능

적합한 상황:
- 특정 함수 리팩토링
- 에러 핸들링 추가
- 타입 변환
```

**사용 예시:**
```
선택: function calculate(a, b) { return a + b; }
프롬프트: "async/await 추가하고 에러 핸들링 넣어줘"
```

### ③ Composer/Agent (Cmd+I / Ctrl+I)

```
기능: 멀티파일 작업, 자율적 에이전트 동작
단축키: Cmd+I (Mac) / Ctrl+I (Windows)

특징:
- 여러 파일 동시 수정
- 터미널 명령 실행 가능
- 계획 → 실행 → 검증 사이클

적합한 상황:
- 새 기능 추가 (여러 파일 걸쳐)
- 대규모 리팩토링
- 테스트 작성
```

### 모드 선택 가이드

```
┌─────────────────────────────────────────────────────────────┐
│                    Cursor 모드 선택 흐름도                    │
└─────────────────────────────────────────────────────────────┘

작업 범위가 어느 정도인가?
        │
        ├─── 현재 줄/블록 ──→ Tab (자동완성)
        │
        ├─── 단일 파일, 특정 영역 ──→ Inline Edit (Cmd+K)
        │
        └─── 여러 파일, 복잡한 작업 ──→ Composer/Agent (Cmd+I)
```

## 3.2 Cursor Rules 설정

### Rules란?

> **Rules**: Cursor가 코드를 생성할 때 따라야 할 지침을 정의한 설정 파일

### Rules 종류

```
우선순위 (높음 → 낮음):
1. Local (수동): @ruleName으로 명시적 포함
2. Auto Attached: glob 패턴 매칭으로 자동 적용
3. Project Rules: .cursor/rules/*.mdc 파일
4. User Rules: Cursor Settings > Rules (전역)
```

### Project Rules 설정 예시

**.cursor/rules/backend.mdc:**
```markdown
---
description: Backend API 개발 규칙
globs: ["src/api/**/*.ts", "src/services/**/*.ts"]
---

# Backend 코딩 규칙

## 기술 스택
- TypeScript strict mode
- Express.js
- PostgreSQL with Prisma ORM

## 코드 스타일
- 모든 함수에 JSDoc 주석 필수
- async/await 사용 (Promise chain 지양)
- 에러는 커스텀 Error 클래스 사용

## 패턴
- Repository 패턴으로 데이터 접근
- Service 레이어에서 비즈니스 로직 처리
- Controller는 요청/응답 처리만

## 금지 사항
- any 타입 사용 금지
- console.log 대신 logger 사용
- 하드코딩된 설정값 금지 (환경변수 사용)
```

### User Rules 예시 (전역)

```
Cursor Settings > Rules:

당신은 시니어 풀스택 개발자입니다.
- 코드 변경 전 항상 이유를 설명하세요
- 잠재적 버그나 성능 이슈를 미리 언급하세요
- 테스트 작성을 권장하세요
- 한국어로 주석과 설명을 작성하세요
```

## 3.3 Cursor 실전 팁

### 컨텍스트 관리

```
@ 심볼로 컨텍스트 추가:
- @file: 특정 파일 참조
- @folder: 폴더 전체 참조
- @codebase: 전체 코드베이스 검색
- @docs: 문서 참조
- @web: 웹 검색 (최신 정보)
```

### Agent 모드 Best Practices

```
✅ Do:
- 작업 전 Git 커밋하기 (롤백 대비)
- 한 번에 하나의 기능에 집중
- 명확하고 구체적인 지시
- 생성된 diff 꼼꼼히 리뷰

❌ Don't:
- 너무 큰 작업을 한 번에 요청
- 검증 없이 모든 변경 수락
- 컨텍스트 없이 모호한 요청
- 여러 모델을 중간에 전환
```

### Plan Mode 활용

```
Plan Mode: 코드 변경 전 계획 수립 모드

사용법:
1. Chat 창에서 "plan" 선택 (agent 대신)
2. Cursor가 코드베이스 분석
3. Markdown 형태의 계획서 생성
4. 계획 검토 후 실행 결정

장점:
- "측정 두 번, 자르기 한 번" 원칙
- 계획서가 레포에 저장됨 (artifact)
- Agent가 계획에 따라 일관되게 작업
```

# 4. Claude Code 심화 활용

## 4.1 Claude Code 개요

### Claude Code란?

> **Claude Code**: Anthropic이 개발한 CLI 기반 에이전틱 코딩 도구.
> 터미널에서 자연어로 코드 작업을 수행.

```bash
# 설치
npm install -g @anthropic-ai/claude-code

# 실행
claude

# 또는 직접 명령 실행
claude "이 프로젝트의 구조를 설명해줘"
```

### Cursor vs Claude Code

| 측면 | Cursor | Claude Code |
|-----|--------|-------------|
| **인터페이스** | GUI (IDE) | CLI (터미널) |
| **작업 방식** | 시각적 diff 리뷰 | 텍스트 기반 |
| **적합한 상황** | 일상적 개발, UI 작업 | 대규모 리팩토링, 자동화 |
| **대규모 파일** | 한계 있음 | 강점 (18,000줄+ 처리) |
| **워크플로우** | IDE 중심 | 터미널/스크립트 중심 |

## 4.2 CLAUDE.md 설정

### CLAUDE.md란?

> **CLAUDE.md**: Claude Code가 세션 시작 시 자동으로 읽는 설정 파일.
> 프로젝트 컨텍스트, 규칙, 명령어 등을 정의.

### CLAUDE.md 위치

```
검색 순서:
1. 현재 디렉토리
2. 상위 디렉토리들 (루트까지)
3. 하위 디렉토리들

글로벌 설정: ~/.claude/CLAUDE.md
```

### CLAUDE.md 예시

```markdown
# Project: E-commerce API

## 프로젝트 개요
B2B 전자상거래 플랫폼의 백엔드 API 서버

## 기술 스택
- Node.js 20.x + TypeScript 5.x
- Express.js 4.x
- PostgreSQL 15 + Prisma ORM
- Jest for testing
- Docker for deployment

## 주요 명령어
- `npm run dev`: 개발 서버 실행
- `npm run build`: 프로덕션 빌드
- `npm run test`: 테스트 실행
- `npm run lint`: 린트 검사
- `npm run typecheck`: 타입 체크

## 디렉토리 구조
```
src/
├── controllers/  # HTTP 요청 처리
├── services/     # 비즈니스 로직
├── repositories/ # 데이터 접근
├── models/       # Prisma 스키마
├── middlewares/  # Express 미들웨어
├── utils/        # 유틸리티 함수
└── types/        # TypeScript 타입
```

## 코딩 규칙
- ES Modules (import/export) 사용
- async/await 패턴
- 에러는 AppError 클래스 상속
- 모든 public 함수에 JSDoc 필수

## 브랜치 규칙
- main: 프로덕션
- develop: 개발 통합
- feature/*: 기능 개발
- fix/*: 버그 수정

## 주의사항
- .env 파일은 절대 커밋하지 않음
- 데이터베이스 마이그레이션은 반드시 PR 리뷰 필요
- API 응답 형식은 src/types/api.ts 참조
```

## 4.3 Claude Code 워크플로우

### 기본 워크플로우: RPITCR

```
R: Research (조사)
   "이 코드베이스의 인증 로직이 어디 있어?"
   "logging.py 파일 읽어줘"
        │
        ▼
P: Plan (계획)
   "JWT 인증에서 OAuth로 마이그레이션하려면 어떤 단계가 필요해?"
   (코드 작성하지 말고 계획만)
        │
        ▼
I: Implement (구현)
   "계획대로 구현해줘"
        │
        ▼
T: Test (테스트)
   "구현한 코드에 대한 테스트 작성해줘"
   "테스트 실행해줘"
        │
        ▼
C: Commit (커밋)
   "변경사항 커밋하고 PR 만들어줘"
        │
        ▼
R: Review (리뷰)
   README나 changelog 업데이트
```

### Test-Driven Development (TDD) 워크플로우

```
1. Claude에게 기대 입출력 기반 테스트 작성 요청
   "다음 요구사항에 대한 테스트 먼저 작성해줘: [요구사항]"
        │
        ▼
2. 테스트 실행 (실패 확인)
   "테스트 실행해줘"
        │
        ▼
3. 테스트 통과하는 코드 구현 요청
   "이 테스트를 통과하는 코드 구현해줘"
        │
        ▼
4. 테스트 재실행 (성공 확인)
        │
        ▼
5. 리팩토링
   "테스트 통과 유지하면서 코드 정리해줘"
```

## 4.4 Claude Code 고급 기능

### Custom Slash Commands

```bash
# 프로젝트 명령어 생성
mkdir -p .claude/commands

# 예: /fix-issue 명령어
cat > .claude/commands/fix-issue.md << 'EOF'
GitHub 이슈 #$ARGUMENTS를 분석하고 수정하세요:

1. `gh issue view $ARGUMENTS`로 이슈 내용 확인
2. 관련 코드 탐색
3. 수정 사항 구현
4. 테스트 작성 및 실행
5. 커밋 메시지 작성
6. PR 생성
EOF

# 사용
claude> /fix-issue 123
```

### Sub-agents 활용

```
Sub-agent: 특정 작업에 특화된 하위 에이전트

종류:
- Explore: 코드베이스 탐색 전문 (읽기 전용)
- Plan: 계획 수립 전문
- Task: 특정 작업 수행

사용 예:
claude> "Explore 에이전트로 인증 관련 코드 모두 찾아줘"
claude> "Plan 에이전트로 리팩토링 계획 세워줘"
```

### Permission Modes

```
모드 전환: Shift+Tab

1. Normal Mode: 각 작업에 권한 요청
2. Auto-Accept Mode: 편집 자동 수락 (⏵⏵)
3. Plan Mode: 읽기 전용, 계획만 (⏸)

주의: --dangerously-skip-permissions는
      격리된 환경(Docker 등)에서만 사용
```

# 5. 효과적인 코드 생성 워크플로우

## 5.1 Research → Plan → Implement → Test 사이클

### 전체 흐름

```
┌─────────────────────────────────────────────────────────────┐
│              효과적인 AI 코딩 워크플로우                       │
└─────────────────────────────────────────────────────────────┘

┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ Research │ ──→ │   Plan   │ ──→ │Implement │ ──→ │   Test   │
│  (조사)   │     │  (계획)   │     │  (구현)   │     │ (테스트)  │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
     │                                                   │
     │                                                   │
     └────────────────── 피드백 루프 ─────────────────────┘
```

### 1단계: Research (조사)

```
목적: 문제와 기존 코드 이해

활동:
- 관련 파일 탐색
- 기존 패턴 파악
- 의존성 확인
- 요구사항 명확화

프롬프트 예시:
"@codebase 사용자 인증과 관련된 모든 파일을 찾아서 
 현재 인증 흐름을 설명해줘. 코드는 작성하지 마."
```

### 2단계: Plan (계획)

```
목적: 구현 전 설계 및 계획

활동:
- 수정할 파일 목록 작성
- 단계별 구현 계획
- 잠재적 리스크 식별
- 테스트 전략 수립

프롬프트 예시:
"OAuth 2.0 인증을 추가하려면 어떤 파일들을 수정해야 하고,
 어떤 순서로 진행하면 좋을지 계획을 세워줘.
 아직 코드는 작성하지 마."
```

### 3단계: Implement (구현)

```
목적: 계획에 따라 코드 작성

활동:
- 계획된 순서대로 구현
- 각 단계 완료 후 확인
- Diff 리뷰 및 승인

프롬프트 예시:
"계획의 1단계부터 시작해줘.
 각 파일 수정 후 변경 내용을 보여주고,
 다음 단계로 넘어가기 전에 확인할게."
```

### 4단계: Test (테스트)

```
목적: 구현 검증

활동:
- 단위 테스트 작성/실행
- 통합 테스트 실행
- 수동 검증
- 엣지 케이스 확인

프롬프트 예시:
"방금 구현한 OAuth 로직에 대한 테스트를 작성하고 실행해줘.
 성공/실패 케이스 모두 포함해줘."
```

## 5.2 프로젝트 초기 설정 Best Practices

### AI에게 PRD 기반 질문하기

```

→ 대화를 통해 PRD 보완
→ 완성된 PRD로 구현 시작
```

# 6. AI 생성 코드 검증

## 6.1 왜 검증이 중요한가?

### 현실: AI 생성 코드의 한계

```
통계 (2025년 기준):
- 96%의 개발자가 AI 생성 코드가 완벽하지 않다고 인식
- 25%는 5개 중 1개 제안에 오류 포함
- 42%의 코드가 AI 도움으로 작성됨
- 그러나 48%만이 "항상" 검증한다고 응답

결론: 검증 없이 AI 코드를 사용하는 것은 위험
```

### AI 코드의 주요 문제 유형

```
1. 환각 (Hallucination)
   - 존재하지 않는 패키지/함수 사용
   - 잘못된 API 호출
   - 틀린 문법

2. 컨텍스트 불일치
   - 기존 코드 스타일과 다른 패턴
   - 프로젝트 규칙 무시
   - 호환되지 않는 버전 사용

3. 성능/보안 이슈
   - 비효율적 알고리즘
   - 보안 취약점
   - 메모리 누수 가능성

4. 불완전한 구현
   - 에러 핸들링 누락
   - 엣지 케이스 미처리
   - 테스트 부재
```

## 6.2 AI 코드 검증 체크리스트

### 기본 검증

```
□ 코드가 컴파일/실행되는가?
□ Lint 오류가 없는가?
□ 타입 체크를 통과하는가?
□ 기존 테스트가 통과하는가?
```

### 기능 검증

```
□ 요청한 기능이 구현되었는가?
□ 엣지 케이스가 처리되는가?
□ 에러 핸들링이 적절한가?
□ 입력 유효성 검사가 있는가?
```

### 품질 검증

```
□ 코드 스타일이 프로젝트와 일치하는가?
□ 네이밍이 명확하고 일관적인가?
□ 불필요한 복잡성이 없는가?
□ 주석/문서가 적절한가?
```

### 보안 검증

```
□ 사용된 패키지가 실제로 존재하는가?
□ SQL Injection 등 취약점이 없는가?
□ 민감한 정보가 하드코딩되지 않았는가?
□ 적절한 인증/인가가 적용되었는가?
```

### 성능 검증

```
□ 알고리즘 복잡도가 적절한가?
□ N+1 쿼리 문제가 없는가?
□ 메모리 사용이 적절한가?
□ 불필요한 연산이 없는가?
```

## 6.3 환각(Hallucination) 감지 방법

### 패키지 환각 감지

```javascript
// AI가 제안한 코드:
import { nonExistentFunction } from 'imaginary-package';

// 검증 방법:
// 1. npm/pip 등에서 패키지 존재 확인
// 2. 패키지 문서에서 함수 존재 확인
// 3. 실제 설치 후 import 테스트

// 자동화:
npm install imaginary-package  // 실패하면 환각
```

### API 환각 감지

```python
# AI가 제안한 코드:
response = openai.ChatCompletion.create(
    model="gpt-5-turbo",  # 존재하지 않는 모델
    messages=[...]
)

# 검증 방법:
# 1. 공식 API 문서 확인
# 2. 실제 API 호출 테스트
# 3. 에러 메시지 분석
```

### 통계적 검증

```
연구 결과:
- 오픈소스 모델: 21.7% 환각률
- 상용 모델: 5.2% 환각률
- 43%의 환각이 반복적으로 발생

대응:
- 모든 import 문 수동 검증
- CI/CD에 의존성 검증 추가
- 새 패키지 사용 시 항상 확인
```

## 6.4 효과적인 코드 리뷰 전략

### AI가 생성한 코드 리뷰 시 집중 포인트

```
1. Import 문 검증
   - 모든 패키지가 실제로 존재하는가?
   - 버전이 호환되는가?

2. 타입 안전성
   - any 타입 남용이 없는가?

3. 에러 처리
   - try-catch가 적절히 사용되었는가?
   - 에러 메시지가 유의미한가?

4. 비즈니스 로직
   - 요구사항과 일치하는가?
   - 엣지 케이스가 처리되는가?

5. 기존 코드와의 일관성
   - 네이밍 컨벤션 준수
   - 아키텍처 패턴 준수
```

### AI 활용 코드 리뷰

```
AI를 리뷰어로도 활용:

- 잠재적 버그
- 성능 이슈

[코드]"

주의: AI 리뷰도 100% 신뢰하지 말 것
      최종 판단은 항상 개발자가
```

# 8. 자가 점검 문제

## 개념 이해 확인

### Q1. Cursor의 세 가지 핵심 모드(Tab, Inline Edit, Composer/Agent)의 차이점을 설명하세요.

**정답:**

- **Tab (자동완성)**: 현재 작성 중인 코드의 다음 부분을 예측하여 제안. 가장 빠르고 간단한 방식. 단일/멀티라인 코드 완성에 적합.
- **Inline Edit (Cmd+K)**: 선택한 코드 영역을 자연어 명령으로 수정. 특정 함수나 블록의 정밀한 수정에 적합. Diff 미리보기 제공.
- **Composer/Agent (Cmd+I)**: 여러 파일에 걸친 작업을 자율적으로 수행. 새 기능 추가, 대규모 리팩토링 등 복잡한 작업에 적합. 터미널 명령 실행 가능.

### Q2. CLAUDE.md 파일의 용도와 포함해야 할 내용을 설명하세요.

**정답:**

**용도**: Claude Code가 세션 시작 시 자동으로 읽어서 프로젝트 컨텍스트를 파악하는 설정 파일

**포함 내용**:
- 프로젝트 개요 및 목적
- 기술 스택 (언어, 프레임워크, 데이터베이스 등)

- 디렉토리 구조 설명
- 코딩 규칙 및 컨벤션
- 브랜치/커밋 규칙
- 주의사항 및 특이사항

### Q3. AI 코드 환각(Hallucination)이란 무엇이며, 어떻게 감지할 수 있나요?

**정답:**

**환각이란**: AI가 존재하지 않는 패키지, 함수, API 등을 마치 실제로 있는 것처럼 생성하는 현상

**감지 방법**:
1. **패키지 검증**: npm/pip 등 패키지 매니저에서 실제 존재 여부 확인
2. **API 문서 확인**: 공식 문서에서 함수/메서드 존재 여부 확인
3. **실제 실행**: 코드 실행하여 import 에러 등 확인
4. **CI/CD 통합**: 자동화된 의존성 검증 파이프라인 구축
5. **통계적 접근**: 반복적으로 나타나는 환각 패턴 모니터링 (43%가 반복됨)

---

### Q4. "Research → Plan → Implement → Test" 워크플로우의 각 단계를 설명하세요.

**정답:**

1. **Research (조사)**: 문제와 기존 코드 이해. 관련 파일 탐색, 기존 패턴 파악, 요구사항 명확화. "코드는 작성하지 말고 분석만"
2. **Plan (계획)**: 구현 전 설계. 수정할 파일 목록, 단계별 계획, 리스크 식별, 테스트 전략 수립. "아직 코드 작성하지 않고 계획만"
3. **Implement (구현)**: 계획에 따라 코드 작성. 순차적 구현, 각 단계 완료 후 확인, Diff 리뷰
4. **Test (테스트)**: 구현 검증. 단위/통합 테스트, 엣지 케이스 확인, 수동 검증

### Q5. Vibe Coding과 Human-Agent Engineering의 차이점을 설명하세요.

**정답:**

| 측면 | Vibe Coding | Human-Agent Engineering |
|-----|-------------|------------------------|
| **접근법** | "AI야 해줘" | "이렇게 해줘, 이유는..." |
| **검증** | 결과만 확인 | 과정과 결과 모두 검증 |
| **속도** | 빠름 | 상대적으로 느림 |
| **품질** | 불안정 | 신뢰할 수 있음 |
| **적합성** | 프로토타입 | 프로덕션 |
| **개발자 역할** | 요청자 | 관리자/리뷰어 |

Human-Agent Engineering에서 개발자는 의도를 설계하고 AI의 작업을 관리/감독하는 역할을 수행합니다.

### Q7. AI가 생성한 다음 코드의 잠재적 문제점을 찾으세요:

```javascript
import { validateEmail } from 'email-validator-pro';
import { sendNotification } from '@company/internal-lib';

async function registerUser(email, password) {
  if (validateEmail(email)) {
    const user = await db.users.create({ email, password });
    sendNotification(`New user: ${email}`);
    return user;
  }
}
```

**정답:**

1. **패키지 환각 가능성**: `email-validator-pro`가 실제 존재하는지 확인 필요
2. **에러 핸들링 부재**: DB 작업이나 알림 전송 실패 시 처리 없음
3. **비밀번호 평문 저장**: 해싱 없이 비밀번호를 저장하는 보안 취약점
4. **반환값 불일치**: validateEmail이 false일 때 undefined 반환
5. **비동기 처리 누락**: sendNotification이 비동기라면 await 필요할 수 있음
6. **입력 유효성 검사 부족**: password에 대한 검증 없음

**정답:**

| 측면 | Cursor Rules (.mdc) | CLAUDE.md |
|-----|---------------------|-----------|
