# 2. AI Agent의 이해

## 2.1 AI Agent란 무엇인가?

### 정의

> **AI Agent**: LLM을 핵심 두뇌로 사용하여, 계획을 세우고, 도구를 활용하며, 복잡한 작업을 자율적으로 수행하는 시스템

**비유**: AI Agent는 "매우 유능하지만 경험이 없는 인턴"과 같습니다. 
- 지식은 풍부하지만 실제 환경에 대한 접근이 제한됨
- 명확한 지시와 적절한 도구가 주어지면 복잡한 작업 수행 가능
- 관리자(개발자)의 감독과 피드백이 필요

### Agent vs 일반 LLM 챗봇

| 특성 | 일반 LLM 챗봇 | AI Agent |
|-----|-------------|----------|
| 상호작용 | 단일 질문-응답 | 다단계 작업 수행 |
| 도구 사용 | 없음 | 외부 API, DB, 파일 시스템 등 활용 |
| 자율성 | 낮음 (수동적 응답) | 높음 (능동적 계획 및 실행) |
| 상태 관리 | 대화 컨텍스트만 | 작업 진행 상태, 메모리 관리 |
| 예시 | ChatGPT 기본 대화 | Cursor, Claude Code, GitHub Copilot |

## 2.2 Agent의 핵심 구성 요소

Anthropic이 제시한 Agent 아키텍처의 기본 구성:

```
┌─────────────────────────────────────────────────────────────┐
│                    Augmented LLM                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌───────────────┐                                        │
│   │     LLM       │  ← 핵심 두뇌 (추론, 계획, 의사결정)       │
│   │   (Brain)     │                                        │
│   └───────┬───────┘                                        │
│           │                                                 │
│     ┌─────┼─────┬─────────────┐                            │
│     ↓     ↓     ↓             ↓                            │
│ ┌──────┐┌──────┐┌──────┐┌──────────┐                       │
│ │Memory││Tools ││Search││  Other   │                       │
│ │      ││      ││(RAG) ││Augments  │                       │
│ └──────┘└──────┘└──────┘└──────────┘                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### ① LLM (Brain)

Agent의 핵심 두뇌로서 다음 역할 수행:
- **추론 (Reasoning)**: 문제 분석 및 해결 방법 도출
- **계획 (Planning)**: 작업을 단계별로 분해
- **의사결정 (Decision Making)**: 어떤 도구를 언제 사용할지 판단
- **결과 종합 (Synthesis)**: 도구 실행 결과를 통합하여 최종 응답 생성

### ② Memory (메모리)

Agent가 컨텍스트를 유지하는 방식:

| 메모리 유형 | 설명 | 예시 |
|-----------|-----|-----|
| **Short-term** | 현재 대화/작업의 컨텍스트 | 현재 세션의 대화 기록 |
| **Long-term** | 여러 세션에 걸친 정보 저장 | 사용자 선호도, 과거 작업 기록 |
| **Working** | 현재 작업 중인 상태 정보 | 중간 계산 결과, 작업 진행 상태 |

### ③ Tools (도구)

Agent가 실제 세계와 상호작용하는 수단:

```
도구 유형:
├── 정보 검색 (Retrieval)
│   ├── 웹 검색
│   ├── 데이터베이스 쿼리
│   └── 파일 시스템 접근
├── 액션 수행 (Action)
│   ├── API 호출
│   ├── 이메일 발송
│   └── 파일 생성/수정
└── 계산/분석 (Computation)
    ├── 코드 실행
    ├── 수학 계산
    └── 데이터 분석
```

### ④ Retrieval/RAG (검색 증강 생성)

외부 지식을 가져와 LLM의 응답을 강화:
- 벡터 데이터베이스에서 관련 문서 검색
- 검색된 정보를 프롬프트에 포함
- LLM이 최신/전문 정보를 기반으로 응답

## 2.3 Workflow vs Agent: 중요한 구분

Anthropic은 "agentic systems"을 두 가지로 구분합니다:

### Workflow (워크플로우)

> **정의**: LLM과 도구가 **미리 정의된 코드 경로**를 통해 조율되는 시스템

```
특징:
- 예측 가능한 실행 흐름
- 개발자가 각 단계를 명시적으로 정의
- 높은 신뢰성과 일관성
- 디버깅이 용이

예시: 
사용자 입력 → 감정 분석 → 적절한 템플릿 선택 → 응답 생성
(각 단계가 코드로 정의됨)
```

### Agent (에이전트)

> **정의**: LLM이 **자체적으로 프로세스와 도구 사용을 동적으로 결정**하는 시스템

```
특징:
- 유연한 실행 흐름
- LLM이 다음 단계를 자율적으로 결정
- 복잡한 문제에 적응 가능
- 예측하기 어려운 동작 가능

예시:
"이 버그를 수정해줘" → Agent가 스스로:
1. 관련 파일 탐색
2. 버그 원인 분석  
3. 수정 방법 결정
4. 코드 수정
5. 테스트 실행
(각 단계를 LLM이 동적으로 결정)
```

### 언제 무엇을 사용할까?

| 상황 | 권장 접근법 |
|-----|----------|
| 작업이 명확하고 예측 가능 | Workflow |
| 높은 신뢰성/일관성 필요 | Workflow |
| 작업이 복잡하고 가변적 | Agent |
| 다양한 입력에 유연하게 대응 필요 | Agent |
| 디버깅/감사가 중요 | Workflow |

**Anthropic의 조언:**
> "가장 단순한 솔루션부터 시작하고, 복잡성은 성과가 명확히 개선될 때만 추가하라"

# 3. Function Calling (Tool Use)

## 3.1 Function Calling이란?

### 정의

> **Function Calling**: LLM이 외부 함수/API를 호출할 수 있도록 하는 기능. 
> LLM은 어떤 함수를 어떤 인자로 호출해야 하는지를 **JSON 형태로 출력**하고, 
> 실제 함수 실행은 **별도의 프로그램**이 수행합니다.

### 핵심 이해: LLM은 함수를 "실행"하지 않는다

```
❌ 잘못된 이해:
"LLM이 직접 함수를 호출하고 실행한다"

✅ 올바른 이해:
"LLM은 어떤 함수를 호출해야 하는지 JSON으로 알려주고,
 실제 실행은 우리 코드가 담당한다"
```

**비유**: LLM은 "무엇을 해야 하는지 지시서를 작성하는 역할"이고, 
실제 일은 "지시서를 받은 직원(우리 코드)"이 수행합니다.

## 3.2 Function Calling의 작동 흐름

```
┌──────────────────────────────────────────────────────────────────┐
│                    Function Calling 흐름                         │
└──────────────────────────────────────────────────────────────────┘

1. 사용자 입력
   "서울의 현재 날씨 알려줘"
          │
          ▼
2. LLM에 함수 정의와 함께 전달
   ┌─────────────────────────────────┐
   │ 시스템: 다음 함수를 사용할 수 있음  │
   │ - get_weather(location: string) │
   │                                 │
   │ 사용자: 서울의 현재 날씨 알려줘    │
   └─────────────────────────────────┘
          │
          ▼
3. LLM이 함수 호출 JSON 생성 (실행 X)
   ┌─────────────────────────────────┐
   │ {                               │
   │   "function": "get_weather",    │
   │   "arguments": {                │
   │     "location": "서울"          │
   │   }                             │
   │ }                               │
   └─────────────────────────────────┘
          │
          ▼
4. 우리 코드가 실제 함수 실행
   get_weather("서울") → API 호출 → "맑음, 15°C"
          │
          ▼
5. 함수 결과를 LLM에 다시 전달
   ┌─────────────────────────────────┐
   │ 함수 실행 결과: "맑음, 15°C"     │
   └─────────────────────────────────┘
          │
          ▼
6. LLM이 최종 응답 생성
   "서울의 현재 날씨는 맑음이며, 기온은 15°C입니다."
```

## 3.3 함수 정의 (Function Schema)

LLM에게 사용 가능한 함수를 알려주는 방식:

### JSON Schema 예시

```json
{
  "name": "get_weather",
  "description": "지정된 도시의 현재 날씨 정보를 가져옵니다",
  "parameters": {
    "type": "object",
    "properties": {
      "location": {
        "type": "string",
        "description": "날씨를 조회할 도시 이름 (예: 서울, 부산)"
      },
      "units": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "온도 단위",
        "default": "celsius"
      }
    },
    "required": ["location"]
  }
}
```

### 좋은 함수 정의의 요소

| 요소 | 중요성 | 예시 |
|-----|-------|-----|
| **name** | 명확하고 동작을 설명 | `get_weather` (O), `weather` (X) |
| **description** | LLM이 언제 사용할지 판단 | "현재 날씨 정보를 가져옵니다" |
| **parameters** | 정확한 타입과 설명 | `location: string - 도시 이름` |
| **required** | 필수 vs 선택 파라미터 구분 | `["location"]` |

## 3.4 Function Calling의 활용 사례

### ① 정보 검색

```
사용자: "애플 주가 알려줘"

LLM 출력:
{
  "function": "get_stock_price",
  "arguments": {"symbol": "AAPL"}
}

→ 실시간 주가 정보 반환
```

### ② 액션 수행

```
사용자: "내일 오후 3시에 팀 미팅 일정 잡아줘"

LLM 출력:
{
  "function": "create_calendar_event",
  "arguments": {
    "title": "팀 미팅",
    "datetime": "2025-01-15T15:00:00",
    "attendees": ["team@company.com"]
  }
}

→ 실제 캘린더에 일정 생성
```

### ③ 데이터 추출/변환

```
사용자: "이 텍스트에서 모든 이메일 주소 추출해줘"

LLM 출력:
{
  "function": "extract_emails",
  "arguments": {
    "text": "[사용자가 제공한 텍스트]"
  }
}

→ 정규식으로 이메일 추출하여 구조화된 데이터 반환
```

## 3.5 Function Calling vs 프롬프트만 사용하기

| 측면 | 프롬프트만 | Function Calling |
|-----|----------|-----------------|
| 실시간 데이터 | ❌ 불가능 | ✅ API 연동 가능 |
| 외부 시스템 연동 | ❌ 불가능 | ✅ DB, 이메일 등 |
| 출력 형식 | 텍스트 (파싱 필요) | 구조화된 JSON |
| 복잡한 계산 | ❌ 환각 가능성 | ✅ 정확한 계산 |
| 구현 복잡도 | 낮음 | 중간 |

# 4. Model Context Protocol (MCP)

## 4.1 MCP란 무엇인가?

### 정의

> **MCP (Model Context Protocol)**: AI 애플리케이션이 외부 데이터 소스 및 도구와 
> **표준화된 방식**으로 연결될 수 있도록 Anthropic이 개발한 오픈 프로토콜

### USB-C 비유

```
하드웨어 세계:                    AI 세계:
┌─────────────┐                 ┌─────────────┐
│   USB-C     │                 │     MCP     │
│  (범용 포트) │                 │ (범용 프로토콜)│
└─────────────┘                 └─────────────┘
      │                               │
      ↓                               ↓
모든 기기가 하나의              모든 AI가 하나의
표준 포트로 연결               표준 프로토콜로 도구 연결
```

**이전 방식 (MCP 없이):**
- 각 AI 도구마다 별도의 커스텀 통합 필요
- API마다 다른 인증, 데이터 형식, 에러 처리
- N개의 AI 앱 × M개의 도구 = N×M개의 통합 필요

**MCP 방식:**
- 한 번 MCP 구현하면 모든 MCP 호환 도구 사용 가능
- 표준화된 인터페이스
- N개의 AI 앱 + M개의 도구 = N+M개의 구현만 필요

## 4.2 MCP 아키텍처

### 핵심 참여자

```
┌─────────────────────────────────────────────────────────────────┐
│                        MCP 아키텍처                              │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────┐
│    MCP Host     │  ← AI 애플리케이션 (Claude Desktop, Cursor 등)
│ (AI Application)│
└────────┬────────┘
         │ 생성/관리
         ▼
┌─────────────────┐         ┌─────────────────┐
│   MCP Client    │◄───────►│   MCP Server    │
│   (연결 관리)    │  통신    │  (도구 제공자)   │
└─────────────────┘         └─────────────────┘
                                    │
                            ┌───────┴───────┐
                            ▼               ▼
                    ┌───────────┐   ┌───────────┐
                    │로컬 데이터 │   │외부 서비스 │
                    │(파일, DB) │   │(API, 웹)  │
                    └───────────┘   └───────────┘
```

### 각 구성 요소의 역할

| 구성 요소 | 역할 | 예시 |
|---------|-----|-----|
| **MCP Host** | AI 애플리케이션, 여러 MCP Client 관리 | Claude Desktop, VS Code, Cursor |
| **MCP Client** | MCP Server와의 연결 유지 및 통신 | Host 내부의 연결 관리자 |
| **MCP Server** | 도구, 리소스, 프롬프트를 제공 | GitHub MCP Server, Slack MCP Server |

### 통신 방식 (Transport Layer)

```
┌─────────────────────────────────────────────────────────────────┐
│                      Transport 유형                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. STDIO Transport (로컬)                                      │
│     ├── 같은 머신에서 실행되는 프로세스 간 통신                    │
│     ├── 표준 입출력(stdin/stdout) 사용                          │
│     └── 예: 로컬 파일시스템 MCP Server                          │
│                                                                 │
│  2. Streamable HTTP Transport (원격)                            │
│     ├── 네트워크를 통한 원격 서버 연결                           │
│     ├── HTTP POST + Server-Sent Events                         │
│     └── 예: Sentry MCP Server, 클라우드 서비스                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 4.3 MCP의 3가지 핵심 Primitives

MCP Server가 제공할 수 있는 세 가지 핵심 기능:

### ① Tools (도구)

> LLM이 **실행할 수 있는 함수**

```json
{
  "name": "weather_current",
  "description": "현재 날씨 정보를 가져옵니다",
  "inputSchema": {
    "type": "object",
    "properties": {
      "location": {"type": "string"},
      "units": {"type": "string", "enum": ["metric", "imperial"]}
    },
    "required": ["location"]
  }
}
```

**특징:**
- LLM이 언제 사용할지 **자율적으로 결정** (Model-controlled)
- 실제 액션 수행 (API 호출, 데이터 수정 등)
- Function Calling과 동일한 개념

### ② Resources (리소스)

> AI 애플리케이션에 **컨텍스트 정보를 제공**하는 데이터 소스

```json
{
  "uri": "file:///project/config.json",
  "name": "프로젝트 설정 파일",
  "mimeType": "application/json"
}
```

**특징:**
- 읽기 전용 데이터 (파일 내용, DB 레코드, API 응답)
- 애플리케이션이 언제 가져올지 결정 (App-controlled)
- RAG의 문서와 유사한 역할

### ③ Prompts (프롬프트)

> 일관된 상호작용을 위한 **재사용 가능한 템플릿**

```json
{
  "name": "code_review",
  "description": "코드 리뷰 요청 템플릿",
  "arguments": [
    {
      "name": "code",
      "description": "리뷰할 코드",
      "required": true
    }
  ]
}
```

**특징:**
- 사용자가 선택하여 사용 (User-controlled)
- Few-shot 예시, 시스템 프롬프트 등 포함 가능
- 일관된 품질의 상호작용 보장

### Primitives 비교 요약

| Primitive | 제어 주체 | 용도 | 예시 |
|-----------|---------|-----|-----|
| **Tools** | Model (LLM) | 액션 실행 | API 호출, 파일 수정 |
| **Resources** | Application | 컨텍스트 제공 | 파일 내용, DB 데이터 |
| **Prompts** | User | 상호작용 템플릿 | 코드 리뷰 템플릿 |

# 5. Agent 워크플로우 패턴

Anthropic의 "Building Effective Agents" 연구에서 제시한 검증된 패턴들입니다.

## 5.1 기본 원칙

> **"가장 단순한 솔루션부터 시작하고, 복잡성은 성과가 명확히 개선될 때만 추가하라"**

```
복잡도 스펙트럼:

단순 ◄──────────────────────────────────► 복잡

Prompt → Prompt   → Workflow    → Full
 Only    Chaining   Patterns      Agent
```

## 5.2 Workflow 패턴 5가지

### 패턴 1: Prompt Chaining (프롬프트 체이닝)

> 각 LLM 호출이 이전 호출의 출력을 처리하는 순차적 흐름

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ Step 1  │ ──→ │ Step 2  │ ──→ │ Step 3  │
│문서 요약 │     │번역     │     │포맷팅   │
└─────────┘     └─────────┘     └─────────┘
```

**적합한 상황:**
- 작업을 명확한 순차 단계로 분해 가능
- 각 단계에서 품질 검증 필요
- 이전 단계 결과가 다음 단계 입력이 되는 경우

**예시:**
1. 긴 문서 요약 → 2. 영어로 번역 → 3. 마크다운 포맷팅

### 패턴 2: Routing (라우팅)

> 입력을 분류하여 적절한 후속 처리 경로로 보내는 패턴

```
                    ┌─────────────┐
                    │   Router    │
                    │ (분류/판단)  │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
     ┌──────────┐   ┌──────────┐   ┌──────────┐
     │ 경로 A   │   │ 경로 B   │   │ 경로 C   │
     │(간단 작업)│   │(복잡 작업)│   │(전문 작업)│
     └──────────┘   └──────────┘   └──────────┘
```

**적합한 상황:**
- 입력 유형에 따라 다른 처리가 필요
- 비용/성능 최적화 (쉬운 작업은 작은 모델로)
- 전문화된 처리 필요

**예시:**
- 쉬운 질문 → Haiku (빠르고 저렴)
- 복잡한 질문 → Sonnet (더 정교함)
- 코딩 질문 → 코드 전문 프롬프트로 라우팅

### 패턴 3: Parallelization (병렬화)

> 작업을 독립적인 하위 작업으로 분할하여 동시 처리

```
                    ┌─────────────┐
                    │   입력      │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
     ┌──────────┐   ┌──────────┐   ┌──────────┐
     │ Worker 1 │   │ Worker 2 │   │ Worker 3 │
     │  (동시)   │   │  (동시)   │   │  (동시)   │
     └─────┬────┘   └─────┬────┘   └─────┬────┘
           │               │               │
           └───────────────┼───────────────┘
                           ▼
                    ┌─────────────┐
                    │  Aggregator │
                    │  (결과 종합) │
                    └─────────────┘
```

**두 가지 변형:**

**① Sectioning (분할)**: 작업을 독립적 부분으로 나눔
- 여러 문서를 동시에 분석
- 이미지의 다른 측면을 동시에 평가

**② Voting (투표)**: 같은 작업을 여러 번 수행하여 합의
- 코드 취약점 검사를 여러 프롬프트로 동시 수행
- 다수결로 최종 판단

**적합한 상황:**
- 하위 작업이 독립적
- 속도가 중요
- 다양한 관점이 필요

### 패턴 4: Orchestrator-Workers (오케스트레이터-워커)

> 중앙 LLM이 동적으로 작업을 분해하고 워커에 위임

```
                    ┌─────────────────┐
                    │  Orchestrator   │
                    │ (작업 분해/조율) │
                    └────────┬────────┘
                             │ 동적 위임
           ┌─────────────────┼─────────────────┐
           ▼                 ▼                 ▼
     ┌──────────┐      ┌──────────┐      ┌──────────┐
     │ Worker A │      │ Worker B │      │ Worker C │
     └─────┬────┘      └─────┬────┘      └─────┬────┘
           │                 │                 │
           └─────────────────┼─────────────────┘
                             ▼
                    ┌─────────────────┐
                    │   Synthesizer   │
                    │   (결과 종합)    │
                    └─────────────────┘
```

**Parallelization과의 차이:**
- Parallelization: 하위 작업이 **미리 정의됨**
- Orchestrator-Workers: 하위 작업을 **동적으로 결정**

**적합한 상황:**
- 필요한 하위 작업을 미리 예측할 수 없음
- 입력에 따라 작업 범위가 크게 달라짐
- 복잡한 다단계 작업

**예시:**
- 코딩: 버그 수정 시 어떤 파일들을 수정해야 할지 미리 알 수 없음
- 연구: 검색 결과에 따라 추가 검색 필요 여부 결정

### 패턴 5: Evaluator-Optimizer (평가자-최적화자)

> 하나의 LLM이 생성하고 다른 LLM이 평가하여 반복 개선

```
     ┌─────────────────────────────────────────────┐
     │                                             │
     │    ┌──────────────┐    ┌──────────────┐    │
     │    │  Generator   │───►│  Evaluator   │    │
     │    │   (생성)      │    │   (평가)      │    │
     │    └──────────────┘    └───────┬──────┘    │
     │           ▲                    │           │
     │           │    피드백          │           │
     │           └────────────────────┘           │
     │                                             │
     │         (평가 통과할 때까지 반복)              │
     └─────────────────────────────────────────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   Final Output  │
                    └─────────────────┘
```

**적합한 상황:**
- 명확한 평가 기준이 존재
- 반복적 개선이 품질 향상에 효과적
- 인간의 피드백 과정을 모방해야 할 때

**예시:**
- 문학 번역: 번역 후 뉘앙스 평가 → 수정 반복
- 코드 생성: 코드 작성 → 코드 리뷰 → 수정 반복
- 글쓰기: 초안 작성 → 편집 피드백 → 수정

## 5.3 Full Agent 패턴

위의 워크플로우 패턴들을 넘어서, **진정한 Agent**는:

```
특징:
1. 사용자와의 대화나 명령으로 작업 시작
2. 독립적으로 계획하고 실행
3. 각 단계에서 환경으로부터 피드백 수집 (도구 실행 결과, 코드 실행 등)
4. 필요시 사용자에게 추가 정보/판단 요청
5. 작업 완료까지 자율적으로 진행
```

**Agent가 적합한 경우:**
- 오픈엔디드 문제 (정해진 단계가 없음)
- 신뢰할 수 있는 환경 피드백이 가능
- 에러 복구가 중요
- 복잡한 의사결정이 필요

**예시:**
- **코딩 Agent**: SWE-bench 작업 수행 (여러 파일 수정)
- **Computer Use Agent**: 컴퓨터를 사용해 작업 수행
- **연구 Agent**: 웹 검색, 문서 분석, 보고서 작성

## 5.4 Agent 설계 3대 원칙

Anthropic이 제시한 효과적인 Agent 구현 원칙:

### 원칙 1: 단순성 유지

```
❌ 피해야 할 것:
- 처음부터 복잡한 멀티에이전트 시스템 구축
- 불필요한 추상화 레이어 추가
- 프레임워크에 과도하게 의존

✅ 권장하는 것:
- 단일 프롬프트로 시작
- 필요할 때만 복잡성 추가
- 기본 LLM API 직접 사용 고려
```

### 원칙 2: 투명성 확보

```
- Agent의 계획 단계를 명시적으로 보여주기
- 각 도구 호출과 결과를 로깅
- 의사결정 과정을 추적 가능하게 설계
```

### 원칙 3: 에러 처리 신중히

```
- 도구 실행 실패에 대한 graceful handling
- 무한 루프 방지 (최대 반복 횟수 설정)
- 사용자에게 개입 요청할 수 있는 메커니즘
```

## 6.4 Agent 디버깅 체크리스트

### 도구 정의 점검

- [ ] 함수 이름이 명확하고 동작을 설명하는가?
- [ ] description이 LLM이 "언제" 사용할지 판단하기에 충분한가?
- [ ] 파라미터 타입과 설명이 정확한가?
- [ ] required 필드가 올바르게 지정되었는가?

### 실행 흐름 점검

- [ ] 도구 호출 결과가 LLM에 올바르게 전달되는가?
- [ ] 에러 상황이 적절히 처리되는가?
- [ ] 무한 루프 방지 장치가 있는가?
- [ ] 각 단계의 로깅이 충분한가?

### 품질 점검

- [ ] Agent가 적절한 도구를 선택하는가?
- [ ] 불필요한 도구 호출이 없는가?
- [ ] 최종 응답이 사용자 질문에 답하는가?
- [ ] 환각(hallucination)이 발생하지 않는가?

# 7. 핵심 용어 정리

| 용어 | 정의 | 비유 |
|-----|-----|-----|
| **AI Agent** | LLM을 핵심으로 계획, 도구 사용, 자율적 작업 수행하는 시스템 | 유능하지만 경험 없는 인턴 |
| **Function Calling** | LLM이 외부 함수 호출을 JSON으로 지시하는 기능 | 지시서 작성자 (실행은 별도) |
| **Tool Use** | Function Calling의 다른 표현 | 도구 사용 능력 |
| **MCP** | Model Context Protocol. AI와 도구 연결 표준 | AI 세계의 USB-C |
| **MCP Host** | MCP Client를 관리하는 AI 애플리케이션 | Claude Desktop, Cursor |
| **MCP Client** | MCP Server와 연결을 유지하는 컴포넌트 | 연결 관리자 |
| **MCP Server** | 도구/리소스/프롬프트를 제공하는 프로그램 | GitHub Server, Slack Server |
| **Tools (MCP)** | LLM이 실행할 수 있는 함수 | 망치, 드라이버 등 공구 |
| **Resources (MCP)** | 컨텍스트를 제공하는 읽기 전용 데이터 | 참고 문서, 설계도 |
| **Prompts (MCP)** | 재사용 가능한 상호작용 템플릿 | 표준 작업 지침서 |
| **Workflow** | 미리 정의된 경로로 LLM/도구 조율 | 조립 라인 |
| **Prompt Chaining** | 순차적 LLM 호출, 각 출력이 다음 입력 | 이어달리기 |
| **Routing** | 입력 분류 후 적절한 처리 경로로 분배 | 우편물 분류 |
| **Parallelization** | 독립적 하위 작업 동시 처리 | 병렬 조립 라인 |
| **Orchestrator-Workers** | 중앙 LLM이 동적으로 작업 분배 | 프로젝트 매니저 + 팀원들 |
| **Evaluator-Optimizer** | 생성-평가 반복으로 품질 개선 | 작가 + 편집자 |
| **ReAct** | Reasoning + Acting. 추론과 행동을 번갈아 수행 | 생각하고 행동하기 |
| **JSON-RPC** | MCP가 사용하는 통신 프로토콜 | 표준 메시지 형식 |

# 8. 자가 점검 문제

## 개념 이해 확인

### Q1. AI Agent의 핵심 구성 요소 4가지를 나열하고 각각의 역할을 설명하세요.

**정답:**

1. **LLM (Brain)**: 추론, 계획, 의사결정을 담당하는 핵심 두뇌
2. **Memory**: 대화 컨텍스트 및 작업 상태 유지 (단기/장기 메모리)
3. **Tools**: 외부 세계와 상호작용 수단 (API, DB, 파일 시스템 등)
4. **Retrieval/RAG**: 외부 지식을 검색하여 LLM 응답 강화

---

### Q2. Function Calling에서 "LLM은 함수를 실행하지 않는다"는 말의 의미를 설명하세요.

**정답:**

LLM은 어떤 함수를 어떤 인자로 호출해야 하는지를 JSON 형태로 "출력"만 합니다. 실제 함수 실행은 개발자가 작성한 별도의 코드가 담당합니다. LLM은 "지시서 작성자" 역할이고, 실제 일을 수행하는 것은 우리 프로그램입니다.

---

### Q3. Workflow와 Agent의 차이점을 설명하세요.

**정답:**

- **Workflow**: LLM과 도구가 **미리 정의된 코드 경로**를 통해 조율됨. 예측 가능하고 일관된 실행 흐름.
- **Agent**: LLM이 **자체적으로 프로세스와 도구 사용을 동적으로 결정**함. 유연하지만 예측하기 어려운 동작 가능.

핵심 차이는 **자율성의 정도**입니다.

---

### Q4. MCP의 3가지 핵심 Primitives(Tools, Resources, Prompts)의 차이점을 설명하세요.

**정답:**

| Primitive | 제어 주체 | 용도 |
|-----------|---------|-----|
| **Tools** | Model (LLM이 자율 결정) | 실행 가능한 함수/액션 |
| **Resources** | Application | 읽기 전용 컨텍스트 데이터 |
| **Prompts** | User (사용자 선택) | 재사용 가능한 상호작용 템플릿 |

---

### Q5. Parallelization 패턴의 두 가지 변형(Sectioning, Voting)을 설명하세요.

**정답:**

- **Sectioning (분할)**: 작업을 독립적인 부분으로 나누어 동시 처리. 예: 여러 문서를 동시에 분석
- **Voting (투표)**: 같은 작업을 여러 번 수행하여 다수결로 결과 결정. 예: 코드 취약점을 여러 프롬프트로 검사 후 다수결

---

## 실전 적용 문제

### Q6. 다음 상황에 적합한 워크플로우 패턴을 선택하고 이유를 설명하세요:

"고객 문의를 받아서 (1) 긴급도 판단, (2) 긴급 문의는 즉시 담당자 연결, 일반 문의는 FAQ 검색 후 응답"

**정답:**

**Routing 패턴**이 적합합니다.

이유:
- 입력(고객 문의)을 분류(긴급/일반)해야 함
- 분류 결과에 따라 다른 처리 경로 필요
- 긴급: 담당자 연결 경로
- 일반: FAQ 검색 → 응답 생성 경로

### Q7. 다음 Function Calling 정의의 문제점을 찾고 개선하세요:

```json
{
  "name": "data",
  "description": "데이터 처리",
  "parameters": {
    "type": "object",
    "properties": {
      "input": {"type": "string"}
    }
  }
}
```

**정답:**

**문제점:**
1. `name`이 너무 모호함 ("data"가 무엇을 하는지 불명확)
2. `description`이 불충분 ("데이터 처리"만으로는 LLM이 언제 사용할지 판단 불가)
3. `parameters`의 `input` 설명 없음
4. `required` 필드 누락

**개선된 버전:**
```json
{
  "name": "analyze_sales_data",
  "description": "판매 데이터를 분석하여 기간별 매출 추이와 상위 제품을 반환합니다. 매출 분석, 판매 현황 질문 시 사용하세요.",
  "parameters": {
    "type": "object",
    "properties": {
      "period": {
        "type": "string",
        "description": "분석 기간 (예: '2024-01', '2024-Q1', '2024')"
      },
      "category": {
        "type": "string",
        "description": "제품 카테고리 (선택사항, 미지정시 전체)"
      }
    },
    "required": ["period"]
  }
}
```

### Q8. "코드 리뷰 자동화 시스템"을 설계한다면 어떤 워크플로우 패턴 조합을 사용하겠습니까? 흐름을 설명하세요.

**정답:**

**Parallelization + Evaluator-Optimizer 조합**

**흐름:**

1. **Parallelization (Sectioning)**:
   - 코드를 여러 관점에서 동시 분석
   - Worker 1: 버그/에러 가능성 검사
   - Worker 2: 성능 이슈 검사
   - Worker 3: 보안 취약점 검사
   - Worker 4: 코드 스타일/가독성 검사

2. **Aggregator**:
   - 4개 Worker의 결과를 종합하여 리뷰 초안 생성

3. **Evaluator-Optimizer**:
   - Generator: 종합된 피드백을 기반으로 리뷰 코멘트 작성
   - Evaluator: 리뷰가 건설적이고 명확한지 평가
   - 평가 통과할 때까지 반복 개선

4. **최종 출력**:
   - 정제된 코드 리뷰 결과 반환
